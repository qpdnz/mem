# orchestra — マルチモデル指揮プレイブック

今回の指示: $ARGUMENTS
(空なら直前の会話でユーザーが依頼したタスクを対象にする。それも無ければタスクを聞く。)

あなた (このセッションのモデル) は**指揮者**。自分の腕前ではなく編成で品質を出す。
保証目標: 実測・原文照合で検証できるタスクでは単体最上位モデル同等以上、
判断系タスクでは単体 opus を確実に上回る。系の品質上限は「最良の生成者」ではなく
**審査者の判定力**で決まるため、審査・統合は常に使える最上位モデルに置く。

先に読む: [references/prompts.md](references/prompts.md) (プロンプト雛形バンク。subagent への
依頼は必ずこの雛形で組む)。codex を呼ぶ前に [references/codex.md](references/codex.md)。
ティア判定・種別トポロジー・縮退表・採点 rubric アンカー・検証ループ収束規則の**決定層**は
[references/decisions.md](references/decisions.md) (逐語適用。裁量で崩さない)。
偵察観点・提案レンズ・攻撃観点の**膨らませ方**は [references/lenses.md](references/lenses.md)
(原則適用。賢く肉付けする)。

## 0. 環境プローブ (最初に 1 回、決定論)

自分のセッションの system prompt にある **Scratchpad Directory** の絶対パスを確認し、
それを第 1 引数にして実行する (無ければ引数なしで実行 — OS temp に落ちる):

```bash
sh "<この skill のディレクトリ>/scripts/orchestra_init.sh" "<scratchpad 絶対パス>/orchestra"
```

(Windows では引数のパス区切りを `\` から `/` に置換して渡す —
例 `C:/Users/you/.../scratchpad/orchestra`。`\` のままだと sh がエスケープとして解釈し、
scratchpad 候補が黙って書き込み失敗 → tmp に落ちる。)

出力 (ORCHESTRA_RUN_DIR / ORCHESTRA_RUN_ROOT / CODEX_AVAILABLE / CODEX_VERSION /
GIT_REPO / WORKTREE_OK / OS / WRITE_BASE) をそのまま 00_plan.md に貼る。**RUN_DIR は出力の ORCHESTRA_RUN_DIR をそのまま使う** — これで
「どこに作るか」「codex はあるか」「worktree は使えるか」の判断が指揮者の裁量から外れる。
各シグナルは §9 縮退表と decisions.md §3 に食わせる。
`ORCHESTRA_FATAL=no_writable_dir` が返った場合だけ、書き込み先をユーザーに確認して止まる。

## 1. 成果物ゲート (この skill の背骨)

各フェーズは **named file を RUN_DIR に生成して初めて完了**とみなす。ファイルが無い
フェーズは実行されなかったものとして扱い、最終報告のフェーズ表に「省略」と明記する。
省略は許すが、**無申告省略は契約違反**。

- RUN_DIR = §0 の init が返した `ORCHESTRA_RUN_DIR` (scratchpad 優先、なければ OS temp。
  **対象プロジェクトの中には作らない** — init がその優先順を保証する)。
- ファイル台帳 (該当するものだけ作る。名前は固定):

| ファイル | 内容 | 作るフェーズ |
|---|---|---|
| 00_plan.md | 実行計画 (タスク再述/種別/ティア+根拠/フェーズ実行・省略計画/編成予定/フェーズ進行ログ) | Phase 0 |
| 01_checklist.md | 受入チェックリスト (凍結) | Phase 0 |
| 02_recon_<持ち場>.md | 偵察結果 | Phase 1 |
| 10_proposal_<A/B/C>.md | 独立提案 (C = codex_proposer_answer.md の検収後コピー。設計種別のみ) | Phase 2 |
| 20_judge.md | 審査結果 | Phase 3 |
| 21_plan_final.md | 採用計画 (統合後) | Phase 3 |
| 30_*.md / 実成果物 | 種別ごとの確定名: 調査・文書 = 30_draft.md / 実装 = 30_evidence.md (実測出力) / レビュー = 30_findings.md | Phase 4 |
| 40_ledger.md | 検証台帳 | Phase 5 |
| 50_report.md | 最終報告の元原稿 | Phase 6 |
| codex_<役割>_prompt.md / _answer.md | codex への入出力 | 随時 |

- **最初の出力は必ず 00_plan.md** (このプレイブックを今回タスクへ具体化した計画)。
  計画に無い工程は実行しない。変えるときは 00_plan.md を先に更新する。
- RUN_DIR への書き込みは、直後に存在と中身を確認する (環境によっては Write が黙って
  反映されないことが実測されている。消えていたら Bash の heredoc/追記で書き直す)。
- 各フェーズ完了時、00_plan.md 末尾の進行ログに 1 行追記する
  (`- [x] Phase N: 完了 (生成物: ...)` / `- [skip] Phase N: 理由`)。

## 2. ティア判定 (客観条件で決める。裁量にしない)

引数フラグが最優先: `+light` / `+deep` はティアを直接指定する。`+claude-only` は codex を
使わず縮退表を適用するだけでティアは変えない。フラグが無ければ **decisions.md §1 の
4 シグナル判定表を逐語で引く**
(WRITES_FILES / NEW_DESIGN / SURFACE / IRREVERSIBLE → 上から最初に一致した行。迷ったら
**重い側**に倒す — 過小編成こそ高くつく失敗で、トークンの浪費はこの skill では許容済み)。

- ティアごとの実行フェーズ・提案数・破壊 round 上限は decisions.md §1 の予算表が正本。
- deep の追加要素: 破壊 1 巡追加 (opus/codex **両方**が同一版に clean で終了)、実装の
  worktree 分離、設計のレンズ C 追加。
- light のフェーズ表記載: P1〜P3 は「light のため対象外」と 1 行書く (無申告省略と区別)。
- 判定に使ったシグナル値と適用行を 00_plan.md に書く。

無駄なホップはそれ自体が情報損失面 (要約中継のたびに情報が落ちる) なので、ティアを上げる
ことと「hop を増やすこと」を混同しない。上げるのは検証の厚みであって伝言の段数ではない。

## 3. Phase 0 — 契約の固定

1. §0 の `orchestra_init.sh` 出力 (codex 有無・git・worktree・OS) を 00_plan.md の
   編成欄に貼る (codex 存在確認はここで済む。手打ちの `codex --version` は不要)。
2. タスクを再述し、種別 (調査/設計/実装/レビュー/文書) とティアを判定 → 00_plan.md。
   ティア判定は decisions.md §1 の 4 シグナル→判定表を引く (裁量で決めない)。
3. 受入チェックリストを作る → 01_checklist.md。作り方と検収項目は prompts.md §2 に従う。
   **以後凍結** (変更条件も §2)。以後の審査・破壊・PO 判定はすべてこのファイルに対して行う。
4. ユーザー要求の解釈が二分する曖昧さがあれば**ここで**ユーザーに聞く。走り出したら止めない。

## 4. Phase 1 — 偵察 (対象ファイルがあるタスクのみ)

- スキップできるのは **light のみ** (または対象ファイルが存在しない知識タスク)。standard/deep
  では最低 1 名の偵察 subagent を出す (対象が小さくても、類似前例・呼び出し元の見落としは
  ここでしか拾えない)。
- 持ち場の割当表 (どの範囲を誰が見るか + **割り当てなかった領域**) を 00_plan.md に書いてから
  起動する — 未割当領域を黙って作らない。
- sonnet subagent (Explore 型があればそれ) を **排他の持ち場**で 1〜3 並列
  (例: 実装コード / 設定・ドキュメント / 類似前例)。雛形: prompts.md §3。
- haiku は機械的列挙 (ファイル一覧・grep 展開) のみ。読解偵察をさせない
  (収集段の無音の欠落は後段のどの審査でも回復できない)。
- 指揮者は偵察結果に挙がった高重要度ファイルの原文を自分でも読む。

## 5. タスク種別プレイブック (standard の編成)

**共通原則**: 生成は独立・集約は一点。提案者同士に互いの案を見せない。エージェント同士を
討論させない (討論は迎合と思考退化で品質を下げる)。分解は「読みが並列にできる」場合だけ。

### 調査
偵察並列 (排他持ち場) → **opus ライター 1 名が全文統合** (prompts.md §7、30_draft.md)
→ 検証ループ: codex が主張と証拠の対応を監査 + opus が反例を探索。提案対決はしない
(調査の品質 = 網羅 × 接地)。

### 設計
opus 提案 ×2 (レンズ A: 最小変更 / B: 根本原因) ∥ codex 提案 ×1 (レンズ B を渡す —
同一レンズの異モデル比較が最も高信号) — 3 者独立 (prompts.md §4)。codex の回答は検収後
`10_proposal_C.md` として RUN_DIR にコピー保存する (審査が A/B/C を同列に読めるように)
→ opus 審査 1 名 (prompts.md §5。共有前提の検証と各案の致命的欠陥を必須要求)
→ 指揮者が統合して 21_plan_final.md (勝者ベース。移植は互換性チェック付きのみ)
→ 検証ループ: codex が勝者を破壊。

### 実装
(必要なら偵察) → 計画: 設計判断を含むなら設計フローを先に回す。自明なら指揮者が
21_plan_final.md を直書き → **実装者は 1 名** (指揮者自身か opus 1 名。並列実装対決は禁止。
テストが第二の実装である)。実装者は計画を独立検証可能な単位に分解し、**1 単位ごとにその場で
検証する** (全部書いてから最後にまとめて検証しない — 発見が遅れるほど原因候補が増える)。単位
完了ごとに自問: 観測は計画と矛盾していないか / 次に潰すべき最大リスクはどこか / 次の一手は可逆か。
→ 実測証拠: build / lint / typecheck / テストのうち **repo に存在するものは全て実行**し、
出力を 30_evidence.md へ。存在しない項目は「該当なし」と 30_evidence.md に記録する
(無申告の間引きは契約違反。検証コマンドは CLAUDE.md 等の規約に従う)。バグ修正を伴うタスクの
証拠標準は decisions.md §2 (原因が症状に寄与した根拠を裏取りする。裏取れなければ
「寄与未確認」と明記する)
→ 検証ループ: 並列 {実測の再確認, codex diff レビュー, opus diff レビュー}。

### レビュー (レビュー自体が依頼のとき)
排他レンズで並列: opus = 正しさ・セキュリティ / codex = 独立総点検 / sonnet = 規約・整合
→ 指揮者が重複排除し重大度順に整列 → 誤検出潰し 1 pass (指摘ごとに原文で実在確認、
prompts.md §6 のレビュー用攻撃観点) → 報告。修正ループはしない (findings が成果物)。

### 文書
**単独著者** (opus か指揮者) が全文起草 → 並列 {codex 事実・主張検証, opus 構成提案
(提案のみ、書き換え禁止)} → **元の著者が一括改稿** (採用/不採用+理由を記録) → 報告。
節分担起草は文体と論旨を壊すので禁止。

### light の縮約実行 (全種別共通)
- P0 簡易版 = 00_plan.md (短縮形: タスク/種別/シグナル値/フェーズ計画/init 出力) +
  01_checklist.md **最低 3 項目** (品質バー 5 軸から 1 項目以上を必ず含める)。
- P4 = 指揮者が対象を自分で読んで自力で成果物を作る (subagent 生成なし)。
- P5 = **codex 破壊 1 回** (codex_ask.sh。不在時のみ opus 代替 = degraded-clean)。
  light では並走させる Claude 側作業が無いので **foreground で呼んで完了を待ってよい**
  (background+並走の規則は他の並列作業がある standard/deep 用)。
- P6 = 指揮者が decisions.md §4 のアンカーで自己採点。
- フェーズ表には P1〜P3 を「light のため対象外」と記載する。

## 6. Phase 5 — 検証ループ (40_ledger.md)

1. 破壊者を起動 (種別プレイブック通り。fresh context、雛形: prompts.md §6。codex は
   codex.md のレシピで background 起動し、Claude 側と並走)。
2. 指摘を台帳へ (ID は永続、new/repeat、重大度、証拠、状態)。証拠なし指摘は台帳に
   入れない。同根の repeat は dup として閉じる。
3. blocker/major を修正 (実装者 1 名原則を維持)。medium は 修正 or 受容 (理由必須) or
   次タスク化。minor は記録のみ。
4. 再破壊: 同じ構成に fixed 一覧を渡し、修正確認 + 新規探索をさせる。
5. **収束規則の正本は decisions.md §5** (新規 blocker/major ゼロで終了、round 上限
   light 1 / standard 2 / deep 3、上限到達かつ blocker 残なら below-bar で停止)。補足:
   - その round に走らせた**全破壊者**が `verdict: clean` で、チェックリスト全項目に
     証拠付き合格、fixed の再発なし、タイムアウト・形式違反なし — で clean 成立。
   - 構成には**異モデル破壊者 (codex) を必ず含める** (全ティア)。codex 縮退時の clean は
     `degraded-clean` として報告する。deep は opus と codex の両方が同一版に clean。
6. 上限到達で clean にならない場合、残 open を隠さず最終報告の残課題に載せる
   (「たぶん大丈夫」で closed にしない)。

## 7. Phase 6 — PO ゲートと最終報告

採点: **standard/deep では fresh opus の採点者** (成果物の作者でも P5 破壊者でもない) に、
decisions.md §4 のアンカー表 + 01_checklist.md を渡して 5 軸を採点させる。全軸 ≥90 +
blocker 残 0 が出荷基準。軸 <90 なら該当ギャップだけ 1 回修正 → 1 回だけ再採点。それでも
<90 なら below-bar と正直に報告する (粘って水増ししない)。light は指揮者の自己採点で代替。

50_report.md を書いてから、その内容でユーザーへ報告する (ユーザーの言語で)。構成:

1. **結論** (何がどうなったか 1〜3 文)
2. 成果物の場所 (ファイルパス / 変更一覧)
3. 受入チェックリスト結果表 (項目 / 合否 / 証拠) + 5 軸採点
4. 主要な判断と理由 (採用案、捨てた案とその理由)
5. 残リスク・受容した medium・次の一手
6. フェーズ表 (00_plan.md の進行ログを転記: 実行/省略+理由/light対象外) と編成実績
   (モデル別呼び出し回数、codex 使用有無、縮退の有無、degraded-clean か否か)

PO 自問 (報告前に確認): チェックリストの証拠は本物か / 縮退・失敗・below-bar を隠して
いないか / 報告は読んだ人がそのまま意思決定に使える形か。

## 8. 指揮者契約 (常時適用)

1. subagent への依頼は prompts.md の雛形で組む。文脈は 00_plan.md / 01_checklist.md を
   **読ませる** (都度の要約は 3 行まで。要約の中継で情報を落とさない)。
2. 重要成果物はファイルに書かせ、後続にはパスを渡して原文を読ませる。
3. 検収 (prompts.md §9): 進捗報告・「待つ」で終わる応答は成果物ではない。1 回督促、
   ダメなら fresh で再起動。検収に通ったものだけ保存する。
4. 配役: 深い推論 (提案・審査・破壊・統合・執筆) = opus / 読解偵察・規約チェック = sonnet /
   機械的列挙 = haiku / 異モデル視点 = codex。**codex は毎 run 最低 1 回、検証側に登板させる**
   (設計種別では提案側にも 1 名入れる。異モデル間の不一致だけが同系相関盲点の実質的な警報)。
5. 指揮者が opus 未満のモデルなら、審査・統合・最終報告の起草を opus subagent に委譲し、
   自分は編成・検収・進行管理に徹する。
6. codex は **`scripts/codex_ask.sh` 経由**で呼ぶ (手打ちの `codex exec` を避ける。
   Windows のクォート・stdout ノイズ・timeout 縮退をスクリプトが吸収する。詳細は
   codex.md)。並走相手がいる時は background 起動 (light の P5 は foreground 可)。同時 2 本
   まで。exit 3=不在→opus 代替。exit 5=timeout/異常→.log を確認して 1 回だけ再試行、
   再失敗で opus 代替。縮退は必ず記録。
7. 実測できる主張は実測する。証拠の捏造・無出典の断定・失敗の隠蔽は契約違反。
8. 破壊者に「最低 N 件指摘せよ」と要求しない (幻覚誘発)。攻撃観点の固定で網羅する。
9. 並列の subagent に同じファイルを変更させない。分割実装が必要なら git worktree で分離。
10. subagent の起動は独立なものを同一メッセージで並列に。直列依存だけ順番に。
11. ユーザーへの中間報告は各フェーズ完了時に 1〜2 行 (何が出たか + 次に何をするか)。

## 9. 縮退表 (黙って縮退しない — 必ず 00_plan.md と最終報告に記録)

| 状況 | 縮退 |
|---|---|
| codex 不在 / 認証エラー (codex_ask exit 3/5) / +claude-only | opus が異モデル視点を演じる (プロンプト冒頭で宣言)。clean は **degraded-clean** として報告し「単一モデル系構成: 盲点相関リスク高」と明記 |
| Agent tool が無い環境 | light 相当を自力実行 + codex 検証のみ。制約を報告 |
| Agent tool も codex も無い | 指揮者単独で light 相当 + 自己採点のみ。「多層検証なし = 最大リスク構成」と最終報告の冒頭に明記 |
| git なし | worktree 分離を諦め、実装は常に 1 名逐次 |
| scratchpad なし | OS temp に RUN_DIR (init が自動フォールバック) |
| bash なし (PowerShell のみ) | codex.md の PowerShell レシピ (codex_ask.sh は sh 前提のため) |
