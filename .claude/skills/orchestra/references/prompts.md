# orchestra プロンプト雛形バンク

subagent / codex への依頼は、ここの雛形で組む。雛形の指示文を要約・省略・言い換えしない
(実行モデルの性能に依らず同じ品質を出すための逐語バンク)。記法は 2 種類:
- `{{...}}` = **値を埋める変数** (パス・種別・round 数など)。
- `《ブロック名》` = **§1 の該当ブロック本文を逐語で貼る**指示 (変数ではない。要約禁止)。
全プロンプトに《CONTEXT_BLOCK》と《DELIVERY_RULES》、分析系 (偵察/提案/審査/破壊/執筆) には
《CLAIM_RULES》も必ず含める。

## 目次

1. 共通ブロック (CONTEXT_BLOCK / CLAIM_RULES / DELIVERY_RULES)
2. 受入チェックリスト作成ガイド (Phase 0)
3. 偵察テンプレ (Phase 1)
4. 提案者テンプレ (Phase 2)
5. 審査員テンプレ (Phase 3)
6. 破壊者テンプレ + 攻撃観点セット (Phase 5)
7. ライターテンプレ (調査・文書の Phase 4)
8. 検証台帳テンプレ
9. subagent 出力の検収ルール

## 1. 共通ブロック

### CONTEXT_BLOCK (全プロンプトの冒頭に貼る)

伝言ゲーム対策: 文脈は指揮者が都度要約し直さない。全員が同じ原文 (00_plan.md / 01_checklist.md)
を読む。指揮者がプロンプトに書き足してよい今回限りの説明は 3 行まで。

```
## 依頼の背景 (あなたは会話履歴を見えない。以下のファイルが文脈の正本)
- まず次を Read せよ: {{RUN_DIR}}/00_plan.md (タスクと計画), {{RUN_DIR}}/01_checklist.md (受入基準・凍結済み)
  (宛先が codex の場合はこの行を使わない — codex は RUN_DIR を読めないため、指揮者はこの行を
  00_plan.md と 01_checklist.md の本文の逐語貼り込みに置き換える。codex.md「文脈の渡し方」参照)
- 追加の参照成果物: {{RUN_DIR 内の該当ファイル絶対パス一覧 / 「なし」。要約を信用せず原文を Read する}}
- 対象: {{repo ルート絶対パス / 「ファイルなし (知識タスク)」}}
- 検証コマンド (判明分): {{build/test/typecheck/lint の実コマンド / 「不明 (自分で package.json 等から特定してよい)」}}
- 今回の依頼の位置づけ (3 行以内): {{この subagent の役割が計画のどこに当たるか}}
- 出力規約: あなたの最終テキストがそのまま {{保存先ファイル名}} として保存される。
  保存するのは指揮者。あなたは RUN_DIR にファイルを書かない (依頼で明示された場合を除く)。
```

### CLAIM_RULES (提案者・審査員・破壊者・ライターに貼る)

```
## 主張の分類規律
すべての主張を次の 4 種に分類して書く:
- [事実] 実測・原文で確認済み。必ず証拠 (file:line、コマンド出力、引用) を併記する。
- [推論] 事実からの導出。前提となる事実を明示する。
- [判断] トレードオフの選択。判断基準 (受入チェックリストのどの項目に効くか) を明示する。
- [仮説] 未検証。「未検証」と明記し、検証方法を 1 行添える。
証拠のない [事実] 表記、分類なしの断定は違反。読み手はこの分類を信頼して意思決定する。
```

### DELIVERY_RULES (全プロンプトの末尾に貼る)

```
## 出力規律
- あなたの最終テキストがそのまま成果物ファイルになる。人間向けの前置き・後書き・進捗報告・
  「続きは後で」「〜を待つ」で終わることを禁止する。指定の形式で完成品だけを返す。
- 待っている暇はない: 子エージェントや外部の完了を待たず、いま得られる情報で完成させる。
- 分からない点は止まらず、[仮説] として明記して先へ進む。
- 日本語で書く (コード・コマンド・識別子は原語)。
```

## 2. 受入チェックリスト作成ガイド (Phase 0、指揮者が自分で作る)

チェックリストは走る前に固定する合否契約。以後の審査・破壊・PO 判定はすべてこれに対して行う。

作り方:
- 5〜10 項目。各項目は「できた / できていない」が単独で判定可能な文にする。
  悪い例:「品質が高い」。良い例:「複数 worker が同一ジョブを二重取得しない機構が実装され、
  その根拠 (トランザクション or 条件付き UPDATE) を file:line で示せる」。
- 各項目に証拠クラスを付ける: (実測) コマンド出力・ログ / (原文) file:line・引用 / (合議) 審査員+破壊者が同意。
- ユーザー要求から出る項目に加え、品質バー 5 軸 (単純性/効率性/正確性/信頼性/進化性) のうち
  このタスクに関係する軸を 1 項目以上入れる。
- 「絶対にやってはいけない方向」を 1〜3 個明文化する (例: 公開 API の互換性破壊、新規依存の追加)。
- **凍結規律**: 提案・実行開始後の変更は (a) ユーザーの新情報 (b) blocker 発見、のみ。変更したら
  台帳に「checklist 改訂: 理由」を記録する。検証ループ中の追加は major 以上の根拠がある時だけ。

チェックリスト自体の検収 (書いた直後に自分で確認。実質の製品はこのリストの質):
- [ ] 各項目は第三者が合否判定できるか (「高品質」「適切に」等の未定義語が残っていないか)
- [ ] ユーザー要求の中で、どの項目にも紐付かない要求が残っていないか
- [ ] 採点しやすい項目だけ集めて、本丸 (ユーザーが本当に欲しいもの) を外していないか
- [ ] 証拠クラスがそのタスクで現実に取得可能か (実測できない環境で「実測」を要求していないか)

## 3. 偵察テンプレ (Phase 1 — sonnet 並列用)

読解を伴う偵察は sonnet に出す (haiku は機械的列挙 — ファイル一覧・grep 展開など出力の正誤を
機械確認できる作業 — 専用。収集段の無音の欠落は後段のどの審査でも回復できない)。
持ち場は排他に割る (2 人に同じ範囲を見せない。漏れ防止は範囲設計で行う)。

```
《CONTEXT_BLOCK を逐語貼付し {{}} を埋める》
《CLAIM_RULES を逐語貼付》

## 依頼: 偵察 ({{担当範囲: 例「server/ 配下」「設定とドキュメント」「類似実装の前例」。他の担当と重複しない排他範囲}})
この範囲で、上記タスクに関係するファイル・事実を洗い出せ。コードの評価や提案はするな (それは後段の役割)。

出力形式:
### 関連ファイル
| パス | なぜ関係するか (1行) | 重要度 (高/中/低) |
### タスクに効く事実
- [事実] ... (file:line)
### 気になった矛盾・落とし穴 (あれば)
- ...

《DELIVERY_RULES を逐語貼付》
```

### haiku 用: 機械的列挙テンプレ (読解・解釈を含まない作業のみ)

```
## 依頼: 機械的列挙 (解釈・要約・評価を禁止する。生の結果だけを返す)
- 作業: {{例: 「<dir> 配下の *.ts を再帰列挙」「pattern X の grep 結果 (パス:行番号:行) を列挙」}}
- 出力: コマンド実行結果の生の一覧のみ。加工するな。件数が 0 なら「0 件」と書く。
- あなたの最終テキストがそのまま保存される。前置き・説明を書くな。
```

## 4. 提案者テンプレ (Phase 2 — opus / codex、相互に見せない)

レンズは方法論で分ける (名前だけ変えて中身が同じ、を防ぐ):
- レンズ A「最小変更」: 既存構造を保ち、受入を満たす最小の差分を設計する。新規概念の導入は 1 個まで。
- レンズ B「根本原因」: 要求の背後にある構造問題を特定し、再発しない形を設計する。差分量は問わない。
- レンズ C「運用・進化」(deep のみ): 6 ヶ月後の変更容易性・観測可能性・障害時運用を最優先に設計する。

```
《CONTEXT_BLOCK を逐語貼付し {{}} を埋める》
《CLAIM_RULES を逐語貼付》

## 依頼: 独立提案 (レンズ: {{A|B|C の定義文を逐語で貼る}})
他の提案者の存在を意識するな。あなたの単独責任で最良の {{提案|解答|設計|修正計画}} を作れ。
- 変更・検討の起点候補: {{偵察で特定した具体ファイル一覧。偵察を略した場合は指揮者が特定して記入。「なし (知識タスク)」も可}}
対象ファイルは要約に頼らず自分で Read してから書くこと。

出力形式:
### 提案本文
(実装タスクなら変更対象ファイルごとの具体的な変更内容。調査なら結論と根拠。)
### 受入チェックリスト自己評価
| 項目 | 自己判定 (満たす/満たさない/部分) | 根拠 |
### 前提
- [事実]/[仮説] を分類して列挙
### リスクと自己申告の弱点
- この提案が一番批判されそうな点を最低 2 つ、自分で書く
### 100点満点の自己採点と根拠

《DELIVERY_RULES を逐語貼付》
```

## 5. 審査員テンプレ (Phase 3 — opus fresh context)

```
《CONTEXT_BLOCK を逐語貼付し {{}} を埋める》
《CLAIM_RULES を逐語貼付》

## 依頼: 提案審査
候補 (全文を Read せよ。存在するものすべて):
- {{RUN_DIR}}/10_proposal_A.md
- {{RUN_DIR}}/10_proposal_B.md
- {{RUN_DIR}}/10_proposal_C.md {{存在しなければこの行を消す}}
どの案が受入チェックリストを満たすかを項目別に採点し、勝者を 1 つ決めよ。
平均案・折衷案を作ることを禁止する。勝者ベースに、敗者から移植する価値のある部品が
あれば「互換性チェック」付きで提案せよ。

出力形式:
### 項目別採点表
| チェックリスト項目 | A の評価 + 引用根拠 | B の評価 + 引用根拠 | C の評価 + 引用根拠 (C があれば) |
### 勝者と決定理由
勝者: {{A|B|C}}。決定打となった項目と根拠。僅差なら何が決め手か。
### 移植提案 (0 個でも可)
| 敗者の部品 | 勝者に入れる価値 | 互換性チェック: 勝者の前提と衝突しないか / 壊れる不変条件はないか | 採否推奨
### 全提案が共有している前提
全案が同じ前提に乗っている場合、その前提自体が正しいかを検証して書く (同系モデルの
共有盲点はここに出る)。
### 両案共通の欠落 (チェックリストを満たさない点)
### 各案の致命的欠陥
案ごとに最低 1 件、最も深刻な欠陥を挙げる (見つからなければ探索過程を列挙して「なし」と宣言)。

《DELIVERY_RULES を逐語貼付》
```

## 6. 破壊者テンプレ (Phase 5 — opus fresh / codex、並列)

「最低 N 個指摘せよ」とは書かない (幻覚誘発)。攻撃観点を固定し、ゼロなら破壊失敗宣言をさせる。

```
《CONTEXT_BLOCK を逐語貼付し {{}} を埋める》
《CLAIM_RULES を逐語貼付》

## 依頼: 破壊レビュー (round {{N}})
検証対象:
- 成果物: {{成果物ファイルの絶対パス一覧}}
- 実装タスクの場合の差分: 変更ファイル = {{絶対パス一覧}}。差分は `git -C {{repo ルート}} diff {{base 指定があれば}}` を自分で実行して取得せよ (git が使えない環境では変更ファイル全文を Read して検証する)。
- 実測証拠 (指揮者が実行済みの build/test 出力): {{30_evidence.md のパス / 「なし」}}
対象の原文/実コードを必ず自分で Read してから検証する。褒めること・良いところ探しを禁止する。

任務は 3 つ (すべて実施):
1. **受入チェックリストの全項目**を証拠クラス通りに検証し、項目ごとに 合格/不合格+根拠 を出す。
   検証過程を先に書き、判定は後に書く (結論先行は検証の省略を招く)。実測可能な主張は
   最低 1 件、実際にコマンドを実行して反証を試みる
   (宛先が codex の場合の注記 — 指揮者はこの括弧を次の文に置き換える: 「実測は読み取り専用で
   安全なコマンドに限る。build/test の実測は上記 30_evidence.md を検分する。計測系コマンドが
   実行不能なら『実測不能』と明記し、原文照合で代替する」)。
2. **前 round までの指摘一覧** ({{台帳の open/fixed 項目を逐語で貼る。round 1 なら「なし」}}) に
   対し、fixed が本当に直って再発していないかを確認する。
3. 下の攻撃観点で新規欠陥を探す。**観点を 1 つも省略しない**。

攻撃観点 (タスク種別 {{種別}} 用):
{{攻撃観点セットから該当種別のリストを逐語で貼る}}

出力形式 (指摘 1 件ごと):
| # | new/repeat | 重大度 (blocker/major/medium/minor) | 引用 or file:line | 失敗シナリオ (具体的な入力/状況→何が壊れる) | 影響 | 最小修正案 | 確信度 (高/中/低) |
規律:
- **証拠 (引用/file:line/実測) が埋まらない指摘は出力しない** (数を稼ぐための瑣末指摘・憶測指摘は禁止)。
- 前 round と同じ根本原因の指摘には repeat を付ける (repeat は新規発見に数えない)。

最後に必ず verdict を 1 行で宣言する (どちらか一方):
- `verdict: clean` — 新規の blocker/major が 0 かつ チェックリスト全項目が合格。minor/medium
  だけなら clean でよい (ループ継続理由にならない)。clean の場合は**攻撃観点ごとに何を確認し
  何が確認できなかったか**を列挙する (この列挙のない clean は無効)。
- `verdict: non-clean` — 新規 blocker/major があるか、チェックリスト不合格項目がある。

《DELIVERY_RULES を逐語貼付》
```

### 攻撃観点セット (タスク種別ごと、逐語で使う)

**実装:**
- 仕様との突合: 要求と実装のズレ、勝手な仕様追加・省略
- 境界と異常系: 空/0/最大/重複/不正入力、途中失敗時の中間状態
- 並行性: 競合、二重実行、順序依存、共有状態の desync
- 障害と復旧: プロセス死/再起動/タイムアウト後に矛盾が残らないか
- 互換性: 既存の呼び出し元・データ・設定を壊さないか
- 単位と型: ms/s、0/1 起点、境界の <=/<、暗黙変換
- 二重管理: 同じ状態を 2 箇所で持っていないか、責務の重複
- 資源の無限増加: 削除・保持方針のないテーブル/ログ/メモリ/ファイルは長期運用でどう膨らむか
- セキュリティ: 入力信頼、注入、権限、秘密情報の露出
- 原因説明の裏取り (バグ修正時): 修正理由として書かれた原因が症状への寄与を実測/原文で
  示せるか (示せないなら 30_evidence.md に「寄与未確認」と明記されているか。直った"はず"は
  証拠にならない)

**設計:**
- 前提の妥当性: その前提が崩れたら設計全体が崩れる箇所
- 要件網羅: 受入チェックリストで担保されない要求はないか
- 単純化: 削っても受入を満たす要素 (YAGNI)、より単純な代替案
- 進化性: 次に来そうな仕様変更 3 つを想定し、どれで壊れるか
- 運用: 観測できるか、壊れたと気付けるか、復旧手順はあるか
- 整合: 用語・責務・データフローの矛盾

**調査:**
- 主張と証拠の対応: 出典が主張を本当に支えているか (過大解釈)
- 反例探索: 結論に反する事実・出典を能動的に探したか
- 網羅性: 別の検索軸・別の情報源で欠けているものはないか
- 鮮度と信頼性: 情報の日付、一次/二次、利害関係
- 結論の飛躍: [事実] から [判断] への接続に穴はないか

**レビュー (レビュー結果を検証する場合):**
- 見逃し: 別レンズでの再走査 (上の実装観点から 3 つ選んで再適用)
- 誤検出: 各指摘は実在するか、コードの読み違いではないか
- 重大度の妥当性: 過大/過小評価の指摘

**文書:**
- 事実誤り・古い記述、内部矛盾
- 読者視点: 前提知識のギャップ、誤読しやすい表現
- 過不足: 目的に対して欠けている節、削るべき節

## 7. ライターテンプレ (調査の統合・文書の起草 — opus)

文書は**単独著者制**: 1 人が全文を書き、レビュアー (codex 事実検証 + opus 構成提案) は
修正提案だけを返し、改稿は必ず元の著者が一括で行う (節ごとの分担起草・他人による書き換えは
文体と論旨の一貫性を壊すので禁止)。

```
《CONTEXT_BLOCK を逐語貼付し {{}} を埋める》
《CLAIM_RULES を逐語貼付》

## 依頼: 成果物の執筆 (あなたが唯一の著者)
素材: {{偵察・提案・計画ファイルのパス一覧}} (Read せよ)。
素材の結論を鵜呑みにせず、[事実] は原典 (ファイル/出典) に当たって確認してから書く。

要件:
- 読者: {{ユーザー / チーム / 一般}}。結論を先頭に。
- 構成は受入チェックリストの項目が全部埋まる形にする。
- 出典・根拠は本文中に付ける (巻末まとめだけにしない)。
{{改稿 round なら: - レビュー指摘一覧: <パス>。各指摘に 採用/不採用+理由 を付けた改稿記録を末尾に}}

《DELIVERY_RULES を逐語貼付》
```

## 8. 検証台帳テンプレ (RUN_DIR/40_ledger.md)

```
# 検証台帳
## 受入チェックリスト判定 (最新 round)
| 項目 | 判定 | 証拠 | round |
## 指摘一覧
| ID | 重大度 | 状態 (open/fixed/accepted/dup/rejected) | 指摘 (1行) | 証拠 | 出所 | 対応内容 | 再検証結果 |
| I-001 | major | fixed | ... | file:line | breaker-opus r1 | ... | r2 で再発なし確認 |
## checklist 改訂履歴 (原則なし)
## 縮退・打ち切りの記録 (codex 不在、タイムアウト等)
```

規律: ID は永続 (round を跨いで振り直さない)。黙って消さない (却下は rejected + 理由)。
accepted (受容) には理由と、必要なら次タスク化のメモを書く。

## 9. subagent 出力の検収ルール (全フェーズ共通)

まず機械的な生存確認 (中身を読む前に):
- subagent がエラー/無応答で死んだ → 同じ prompt で fresh を 1 回だけ再起動。再失敗なら
  その役割を縮退 (別モデルで代替 or 編成縮小) し、台帳の「縮退・打ち切り」に記録する。
- codex (codex_ask.sh) → exit code を見る: 0 以外、または answer ファイルが存在しない/空
  なら失敗。exit 3 = 不在 → opus 代替。exit 5 = timeout/異常 → .log を確認して 1 回だけ
  再試行、再失敗で opus 代替。いずれも記録必須。

次に内容の検収 (成果物として保存する前に 3 点):
1. **完成品か**: 進捗報告・「待つ」「後で」で終わっていないか。→ 該当したら SendMessage で
   「いま手元の情報で最終形を完成させよ。形式は◯◯」と 1 回だけ督促。それでもダメなら
   新しい subagent を fresh で起動し直す (同じ prompt + 「一発で完成させよ」)。
2. **形式通りか**: 指定した出力形式の節がすべて在るか (破壊者なら verdict 行の有無も)。
   欠けていたら欠けた節だけ追完させる。
3. **証拠があるか**: [事実] に証拠が付いているか。無根拠の断定が核心部分にあれば追完させる。
検収に通った応答だけを RUN_DIR に保存し、台帳・phase ログを更新する。督促・再起動・縮退は
すべて回数ごと記録する (隠れた品質劣化を残さない)。
