Webシステム版・命名ルール

連番 → 役割 → 大項目 → 中項目 → 小項目

大項目は「画面・処理の一番大きい部位」、中項目は 業務ドメイン名、小項目はその中の具体部品・操作です。 これにより、カテゴリ分けされた中で「どの業務機能のどの部品なのか」をファイル名だけで判定できます。

Ui役割 / responsibility
table大項目 / largest part
invoice中項目 / domain
row.tsx小項目 / concrete part

5. 中項目は業務ドメイン名。絞りすぎない

中項目は order / delivery に固定しません。 そのシステムで「画面・API・SQL・VOをまたいで追いたい業務概念」を入れます。

アカウント・権限系
authaccountuserrolepermissiontenantprofile
販売・EC系
ordercartproductcataloginventorycouponreturn
決済・請求系
paymentinvoicebillingsubscriptionrefundcontract
物流・業務運用系
shipmentdeliverywarehousedriverrouteschedule
社内管理・タスク系
projecttaskticketsupportmessagenotificationapproval
分析・監査系
reportanalyticsauditmetriclogexport
判断基準: table や detail は画面部位なので大項目。 order や customer は業務ドメインなので中項目。 row や summary や search は具体対象なので小項目。

判断ルール

画面を作る？

Ui。React/Vue/Svelte component、HTML断片、table row、detail card。

値を束ねる？

Bind。input → criteria、row → VO、result → state。

正しいか確認？

Val。required、range、permission、invariant。

作る？

Fac。SQL、DTO、VO、event、response を生成。

実行する？

Ope。DB、API、更新、送信など副作用あり。

入口？

Sys。route handler、controller、middleware、framework hook。

運用メモ

このHTMLでは、元ルールの「役割で責務を分け、大・中・小でどの画面・部品・操作かを読む」という考え方を、 Webシステム向けに 連番_役割_大項目_中項目_小項目 へ並び替えています。 中項目は業務ドメイン名なので、プロジェクトごとにドメイン辞書を先に決めると命名が安定します。
