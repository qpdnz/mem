Serena トークン消費 比較検証まとめ

サブエージェントで Serena あり/なし のトークン差分を計測し、用途別の損益・プロジェクトのルール／ドキュメント整備度まで分析（2026-06-02 / 短期サンプル）

ひとことで：「Serena という AI 用のコード支援ツールを使うと、トークン（＝AI の処理コスト）は得か損か」を、同じ作業を使う版／使わない版で同時に走らせて比べた記録です。

やり方（これだけ分かればOK）

  • 同じ1つのタスクを、2体のサブエージェント（使い捨ての作業用 AI）へ 並列で同時に 渡す。
  • 片方は Serena あり、もう片方は Serena なし。それ以外の条件はそろえる。
  • 各自が そのタスク中に消費したトークン量 を記録し、引き算で比べる（少ない方が安い＝良い）。
  • 原則は 読み取り専用（コードは書かない）。実装を試す回だけ使い捨てコピーで行う。

知りたいこと：Serena は 毎回使うべきか？ どんな作業で 得をして／損をするのか？

用語（はじめての人向け）
Serena
AI にコードの「シンボル検索」と「プロジェクトの記憶（メモ）」を与える支援ツール。あり＝この道具を使える状態／なし＝ふつうのテキスト検索だけ。
サブエージェント
本体とは別に一時的に立てる作業用 AI。今回は2体を並列で走らせて同条件で比較した。
トークン
AI が読み書きする処理量の単位。多いほどコスト高。「少ない＝安い」が良い。
作業区間差分
開始〜完了でスレッド累計トークンがどれだけ増えたか＝そのタスクの消費量。本文の比較数値はこれ。
あり／なし
同じタスクを Serena を使って解いた版（あり）と、使わずに解いた版（なし）。

1経緯

  • 元々は「作業ごとに消費トークンを記録する」だけのルールだった。
  • そこへ 「あり/なし の比較は毎回やらない」 を追加（毎回2回走らせるとトークンが倍かかるため）。必要なときか、定期サンプルのときだけ比較する。
  • 「今日だけ試したい」となり、同じ調査タスクを2体のサブエージェントへ並列投入して消費トークンを比べる手順を決めた（＝この検証）。

2運用ルール（確定分）

  • 毎回：Serena 利用有無 / 開始累計 / 完了前累計 / 作業区間差分 を記録（厳密なターン最終値でなく、スレッド累計スナップショットの差分として扱う）。
  • たまに：あり/なしの比較サンプルを取る（明示時または定期調査時のみ）。
  • 通常：比較のためだけに同じ作業を2回実行してトークンを倍消費しない。
  • 比較方法：読み取り専用の同一タスクを2サブエージェントへ並列、親スレッドは結果比較だけ。
  • 編集比較は同一ワークスペースで同時変更させず、必要時だけ使い捨てワークツリーを分ける。

6結果の解釈 ― 「Serena が勝った」と言えるのはどれか↑ 目次

明確に Serena が勝った（＝同じ仕事を より少ないトークン で終えた）と言えるのは、実質 横断修正（自己発光の質感生成の共通化） だけ。そのタスクが Serena 向きだったため。

横断修正で勝った理由

  • 対象が エディター / ゲーム / 共有コア にまたがる。
  • しかし見るべき箇所はかなり絞れた（マテリアル計算、シーンオブジェクト生成のエディタ側／ゲーム側）。
  • 記憶（core / conventions / tech_stack）が効き、「共有コア＝純粋ロジック / Bevy 変換層＝Bevy 依存」の境界判断を早くできた。
  • 通常探索側はテキスト検索＋ファイル確認で広めに見に行く必要があり、結果的に多く読んだ可能性が高い。

逆に局所修正（インスペクター）では負けた理由

  • 対象ファイルや分岐がほぼ分かっており、普通にテキスト検索で当てた方が速いタイプ。
  • あり側は記憶読み＋シンボル探索＋診断確認まで実施。品質確認は厚いがトークンは重くなった。

レビュー専用は Serena のツールが使えておらず、比較としては無効。

使い分けの結論
状況判定
小さい既知箇所の修正Serena は重くなりやすい → 使わない
横断的で設計境界が重要な修正Serena が勝つ可能性 → 使う
レビュー今回は判定不能
→ 全作業で使うのでなく、共有コア / エディター / ゲーム をまたぐ・責務境界が重要・対象が複数ファイルに散っているときだけ使うのが合理的。

8Serena に保存されているメモ（AI 記憶フォルダ）↑ 目次

計測③で Serena あり側が読んだのは core conventions tech_stack task_completion。実体は下の6つのメモ。core が起点で、そこから各メモへ参照で辿る構造になっている。

core（起点）プロジェクト地図
  • Windows 上の Rust / Bevy ワークスペース。3つのクレート＝エディター / ゲーム / 共有コア（既定はエディター）。
  • 入口ルール：作業開始時に「AI 入口ルール」を読む。「非公開ノート領域」は探索・参照・編集・コミット禁止。
  • 作業種別ごとの詳細ルール振り分け（命名ルール、エディター設計ルール、コマンド早見表）。
  • 主要入口（エディター入口 / ゲーム入口 / 共有コア）と生成物の場所、Python 生成専用のファイル名プレフィックス。
  • 他メモ（技術スタック / コマンド / 規約 / 完了確認）への参照。
conventions命名・設計規約
  • 報告・回答・コミットメッセージは日本語。非公開ノート領域は常に無視。
  • 場当たり的な hotfix / 暫定対応で完了扱いにせず、根本原因に抜本対応。
  • 命名の正本は「命名ルール」。ファイル名は「役割・機能・対象」を表す規約。既存プレフィックス（ウィンドウ系 / シーンカメラ系 / シーン系）を流用し、新規はコードより先にルールへ追記。
  • メソッドレシーバを使わず処理プレフィックスのモジュール関数へ分離。補助ファイルは親の局所分岐だけに使い、横断ヘルパや純粋計算の置き場にしない。
  • シーンとゲームは分離。Python 生成専用プレフィックスは手書きで使わない。他者の差分を勝手に戻さない（広い git reset/checkout 禁止）。
tech_stack技術スタック
  • Rust edition 2024 / resolver 3。Bevy 0.18（既定機能オフ、エディター/ゲームは動的リンク）。
  • エディター：実行バイナリ1つ、依存に 共有コア / serde / glam / winit / クリップボード(arboard)。
  • ゲーム：実行バイナリ1つ、依存に 物理エンジン(avian3d) / 共有コア / serde。
  • 共有コア：共有の純粋・データ用クレート。
  • ビルド設定：インクリメンタル無効、run/check/build エイリアス、Windows 用リンカ。Bevy ビルド速度向けに調整。
task_completion完了時の確認
  • 完了前に変更状態を確認（非公開ノート領域は除外）し、対象外差分・未追跡を点検。
  • コミット前に staged 一覧を確認。コミットメッセージは日本語。
  • Rust 変更後は必要に応じて整形（cargo fmt）と狭い型チェック。
  • ビルド／チェック前は「コマンド早見表」の競合ガードを確認し、既存のエディター／ゲームを止めない。
  • 画面表示の確認は早見表のスクリーンショット検証方針に従う。
suggested_commands実行コマンド
  • エディター起動（通常入口）/ ゲーム起動 / 各種型チェック / 整形。
  • 競合回避用 script：状態確認（既存プロセス確認）/ 競合回避つきチェック / 隔離ビルド / 起動スモーク（既存ありなら成功扱いでスキップ）。
  • 旧互換エイリアスより、新しい入口（エディター起動 / エディタービルド）を優先。
memory_maintenanceメモ運用方針（Serena 標準）
  • 段階的探索：core を起点に参照でメモのグラフを辿る。参照側がいつ読むかを示す。
  • 書き方：散文でなく密な作業ノート。不変条件と簡潔な箇条書き優先。
  • 追加基準：安定した非自明のプロジェクト規約のみ。汎用知識・一回限りメモ・揮発的な行レベル詳細は入れない。
  • 保守：リネームはツール経由で参照を自動更新。古いメモは点検コマンドで確認。
補足：.pkl はメモではない。 AI 記憶フォルダ内のキャッシュ（.pkl 2ファイル）は、Serena がシンボル探索を速くするための バイナリのシンボル索引キャッシュで、人が読む文章メモではない。実際に読み書きされる「メモ」は上記6つ。

9補足分析 ― エディター / ゲーム のルール非対称↑ 目次

仮説：「エディター系はルールが発達しているが、ゲーム系はルールを一切作っていない。だから差が出た」。実コードベースで確認した。

結論：仮説はほぼ裏づけられた。

観点エディターゲーム
専用の設計ルールあり：「エディター設計ルール」（表題「エディター設計思想」）なし（「ゲーム設計ルール」は存在しない）

私の分析
  • ルール非対称は実在する。エディターには「設計思想＋カメラ / ビューポート / 描画 / 設定値」の明文ルールがあるが、ゲーム実行側に対応する設計ルールは無い。
  • エディター設計ルールの描画規定（UI カメラ・ビューポート・既定カメラ・透明色の扱い）は エディター画面前提。ゲーム画面の Bloom / HDR / 自己発光はどのルールにも書かれていない。
  • ＝ 新しい描画機能は「ルールがあり活発に開発中のエディター側」へ先に入り、ルール無きゲーム実行側が取り残される。今回の「ゲーム画面で自己発光が光らない」は、この構造的ドリフトの典型例。
→ 方向性：ゲーム側の描画（自己発光 / Bloom / HDR / カメラ効果）を 共有コアの共通経路へ寄せ、エディター設計ルールに ゲーム描画の節を足すか、別途ゲーム設計ルールを起こすのが筋。

  • 形式的な doc コメント（rustdoc）はどのクレートも実質ゼロ（0.0〜0.2%）。rustdoc 基準の「整備度」は3つとも ~0%。
  • 理由：このプロジェクトのドキュメントはコード内 doc ではなく、ルール集と AI 記憶（外部）に集約されている。コード内 doc% が全クレート低いのは設計上の選択で、「壊れている」わけではない。
  • 差が出るのは コメント密度。エディター 2.6% ≒ 共有コア 2.6% に対し、ゲームは 0.04%。
  • ＝ ゲームはコード内の説明が事実上皆無で、3つの中で最もドキュメント／コメントが薄い。前節「ゲームはルールも薄い」と完全に一致する。
→ 「整備度」を1指標で言うなら コメント密度が実態を最もよく表す：エディター ≒ 共有コア（約2.6%）≫ ゲーム（約0%）。
