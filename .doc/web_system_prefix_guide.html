Webシステム prefix 命名ルールガイド

大項目の前に連番を置き、大項目は「画面・システムの一番大きい部位名」、中項目は auth / user / value_object / manipulator のようなドメイン機能名にします。これにより、カテゴリ分けされた中で「どの機能のファイルか」を判定しやすくします。

役割prefix 連番 + 大項目 中項目 = ドメイン機能名 小項目 = 具体対象・操作
重要ルール：ファイル名の形
Ui_210_page_auth_login_form.tsx
Ope_240_table_manipulator_row_drag.ts
Bind_230_form_value_object_money_field.ts

1. 基本ルール

役割で責務を読み、連番付き大項目でカテゴリを読み、中項目でドメイン機能を読む。

命名フォーマット

役割prefix_連番_大項目_中項目_小項目[_対象].ext
役割prefixそのファイルが何をするか。例: Ui_, Api_, Val_, Ope_
210_page連番 + 大項目。大項目は一番大きい部位名。例: page, form, table, endpoint
auth中項目。カテゴリ内のドメイン機能名。例: auth, user, value_object, manipulator
login_form小項目。具体対象・UI部品・操作。例: field, row_drag, submit, detail

この形にする理由

  • 1
    連番でカテゴリを固定する
    フォルダや一覧で並んだ時に、同じ部位のファイルを集めやすい。
  • 2
    大項目は最大の部位名にする
    user ではなく page / form / table のように、画面上の大きな場所を置く。
  • 3
    中項目にドメイン機能名を置く
    value_object や manipulator のように、そのカテゴリ内で何の機能かを示す。
  • 4
    小項目で具体対象・操作を絞る
    row, field, drag, preview など、実装箇所を一段詳しくする。

4. 中項目：ドメイン機能名

中項目には「そのカテゴリ内で、どのドメイン機能か」を置く。

中項目の代表セット

authuseraccountprofileorgrolepermissionproductinventoryorderpaymentbillinginvoicenotificationsearchfilereportdashboardsettingauditvalue_objectmanipulatormetadataworkflowimportexport
例: 230_form という入力カテゴリの中で、user のフォームなのか、value_object のフォームなのかを中項目で判定する。

中項目を置いた時の読み方

ファイル名読み方
Ui_230_form_user_profile_field.tsxフォームカテゴリの中にある user 機能の profile field UI
Bind_230_form_value_object_money_field.tsフォームカテゴリの中にある value_object 機能の money field binding

Api_500_endpoint_order_create.tsAPI endpoint カテゴリの中にある order 機能の create API

8. 運用ルール

増やしすぎず、表記ゆれを抑えて、検索しやすい命名にする。

採用基準

  • ✓
    大項目は部位名。page, form, table, endpoint など。
  • ✓
    中項目はドメイン機能名。auth, user, value_object, manipulator など。

避ける語

  • !
    manager, util, helper, common の乱用。
  • !
    ctrl, data, sys など、責務や役割prefixと衝突する語。

変更タイミング

  • →
    新規作成、責務分割、明示リネーム時から適用。
  • →
    既存ファイルは命名規則だけのために一括リネームしない。
