chunks / documents の作り方ルール案

なんでルールがいるの？

いま src/m03_seed.rs は人が手で文を切って入れているだけで、書き方のルールがありません。
書き方がバラバラだと、検索精度がブレるし、将来 AI に自動でチャンク化させるとき(=PIIマスキング後の問い合わせを自動投入する構想)に、何を AI に守らせるかが決まりません。
だから「人が書くときも AI に書かせるときも同じルール」を1つ決めておきます。

chunks 1つを作るルール

RAG 検索は「ヒットした chunk だけ」を AI に渡します。だから chunk 単体で意味が完結していることがすごく大事。

ルール 1. 1 chunk = 「1つの判断 + その対応」。条件と対応はセットで入れる。
✗ 悪い例（条件と対応をバラバラに分けた）
chunk_a: 「顧客がログインできない場合は、」
chunk_b: 「メールアドレスを確認する。」
なぜダメ？ chunk_b だけ取れたとき、いつメールを確認するのか分からない。AIが「メールを確認しましょう」としか答えられない。
○ 良い例（条件と対応をセットで）
chunk: 「顧客がログインできない場合は、メールアドレス・アカウント状態・パスワード再設定履歴を確認する。」
なぜ良い？ 1つ取ってくるだけで「いつ」「何を」が両方分かる。AIが状況に合わせた回答を作れる。
ルール 2. 目安は 50〜200字 / 1〜3文。短すぎず長すぎず。
✗ 悪い例（細かすぎ）
chunk_a: 「ログインできない。」
chunk_b: 「メールを確認。」
chunk_c: 「ロックを確認。」
なぜダメ？ 1チャンクあたりの情報が少なすぎて、ベクトル化しても他の似た短文と区別がつかない(=検索でヒットしない)。
✗ 悪い例（大きすぎ）
chunk: 「ログインできない時のマニュアル。本人確認の流れ。メール確認、アカウント状態、再設定履歴、ロック解除、多要素認証、サポート連絡先、エスカレーション基準、SLA、過去事例…(1000字)」
なぜダメ？ 1チャンクに話題が多すぎて、ベクトルが平均化されて何で検索してもボンヤリ当たる。AI も「何に答えるべきか」迷う。
○ 良い例（ちょうどいい粒度に揃える）
chunk_001: 「顧客がログインできない場合は、メールアドレス・アカウント状態・パスワード再設定履歴を確認する。」
chunk_002: 「アカウントロック状態の場合は、本人確認後に管理画面からロックを解除する。」
chunk_003: 「多要素認証の失敗が3回以上の場合は、本人確認後に二要素設定をリセットする。」
ポイント: どれも「条件 + 対応」が1セット。50〜120字くらい。並べると手順マニュアルとしても読める。
ルール 3. chunk に 個人情報(PII)を入れない。氏名・メール・電話番号・住所などはマスキングしてから chunks に入れる。
✗ 悪い例（PIIそのまま）
chunk: 「[顧客名]([EMAIL])からログイン不可の問い合わせ。パスワード再設定後も入れない。」
なぜダメ？ RAG 検索でヒットすると、別人の対応中に他の顧客の個人情報が AI に渡る。情報漏洩。
○ 良い例（マスキング済み）
chunk: 「顧客からログイン不可の問い合わせ。パスワード再設定後もログインできない場合は、再設定リンクの有効期限と発行履歴を確認する。」
ポイント: 生 PII は raw_inquiries に隔離。chunks には「対応に必要な情報」だけ入れる。
ルール 4. chunk_index は 同じ document 内で 1 から連番。手順の順序を保つ。

documents 1つを作るルール

document は「目次の見出し」のようなもの。chunks の親なので、ここがブレると後で整理できなくなります。

ルール 1. 1 document = 1トピック。1つの Q&A、1つの対応マニュアルに対応する粒度。
✗ 悪い例（詰め込みすぎ）
doc_xxx: 「認証関連全般」
chunks: [ログイン不可, パスワード再設定, 二要素認証解除, 退会, アカウント削除, ...]
なぜダメ？ 検索結果に「認証関連」しか出ないので、結局どの手順か分からない。category と被って情報が二重になる。
○ 良い例（1トピック1ドキュメント）
doc_002: 「ログイン不可の対応」
doc_007: 「パスワード再設定の手順」
doc_010: 「二要素認証の解除手順」
ポイント: タイトルだけ見れば「これは何の手順か」がすぐ分かる粒度。chunks はその中の細かいステップになる。
ルール 2. title は 「対象 + 操作 / 問題」の形に揃える。自由記述しない。
✗ 悪い例（書き手によってブレる）
「対応マニュアルver2」
「顧客対応(2024年最新)」
「メモ」
なぜダメ？ タイトルから内容が読めない。検索しても title hit が役に立たない。
○ 良い例（パターン固定）
「ログイン不可の対応」 ← 対象「ログイン不可」+ 操作「対応」
「請求書宛名の変更手順」 ← 対象「請求書宛名」+ 操作「変更手順」
「退会希望時の対応」 ← 対象「退会希望」+ 操作「対応」
ポイント: 「〜の対応」「〜の手順」「〜の変更手順」など末尾を揃えると並べたとき読みやすい。
ルール 3. category / visibility / status は 決まった値からだけ選ぶ(enum 化)。自由記述しない。

category の決め打ち候補

値意味例
認証ログイン・パスワード・二要素まわりログイン不可、パスワード再設定
請求請求書・支払い・宛名・金額請求書宛名変更、支払い方法変更
解約退会・解約・契約終了退会希望時の対応
設定変更アカウント設定・通知・プロフィールメールアドレス変更
内部社内向け・未承認メモ営業引継ぎメモ

悪い例: 「認証関連」「ログイン系」「auth」「セキュリティ」が混ざる → 同じ意味で4種類できて検索が壊れる。

visibility / status の決め打ち候補

カラム許可する値意味
visibilitycs_only / admin_only / public誰まで検索結果に出していいか
statusdraft / approved / archived承認状態。RAG検索は approved だけ対象にする想定
source_typefaq / memo / inquiry_summary元データの種類。inquiry_summary は AI 要約由来

タグ付け(tags)のルール

category(大分類1個)だけでは「どの製品の話か」「どのシステムの話か」が分からない。
そこで tag(細かいキーワードを複数つけられる) を使って絞り込めるようにする。

category と tag は別物

categorytags
役割 大分類(フォルダ的) 細かいキーワード(複数つけられる)
個数 1ドキュメントに 1個 1ドキュメントに 1〜5個
値の決め方 enum で決め打ち(認証 / 請求 / 解約…) 固定はしないが「よく使うリスト」を共有して揺れを防ぐ
例 "認証" ["レジ", "認証サーバー", "ログイン不可"]

具体例で見る

✗ category だけだと…
doc_a: title="ログイン不可の対応", category="認証"
doc_b: title="ログイン不可の対応", category="認証"
doc_c: title="ログイン不可の対応", category="認証"
問題: 全部「ログイン不可 / 認証」なので、レジ用なのかWeb管理画面用なのか区別できず、検索でどれが当たるか運任せ。
○ tag をつけると区別できる
doc_a: tags=["レジ", "認証サーバー"]
doc_b: tags=["Web管理画面", "二要素認証"]
doc_c: tags=["モバイルアプリ", "パスワード再設定"]
結果: 質問に「レジが…」と書かれていたら doc_a を優先表示できる。誤ヒットが減る。
ルール 1. tag は 1〜5個。多すぎると意味が薄れる。
ルール 2. tag は 短い名詞(製品名 / システム名 / 操作対象)。文にしない。
✗ 悪い tag
["レジでログインができない時の対応について"]
["重要", "至急", "やばい"]
文になっている / 主観的で内容を表していない → 検索の絞り込みに使えない。
○ 良い tag
["レジ", "認証サーバー", "ログイン不可"]
["請求書", "宛名変更", "法人契約"]
短い名詞で、「何に関する手順か」が一目で分かる。
ルール 3. よく使う tag は 共有リストにして揺れを防ぐ。
(例: 「レジ」と「POS」と「レジ端末」が混ざると、検索しても全部当たらない)

よく使う tag リスト案(揺れ防止用)

分野使う tag避ける表記(揺れ)
製品 / システム レジ / 認証サーバー / Web管理画面 / モバイルアプリ / 決済端末 POS, レジ端末, 認証鯖, 管理画面, スマホアプリ
操作対象 ログイン / パスワード / 二要素認証 / アカウントロック / 請求書 / 宛名 サインイン, PW, 2FA, ロック状態, インボイス, 宛先
状況 / 問題 ログイン不可 / 再設定不可 / ロック解除 / 変更依頼 入れない, リセットできない, アンロック, 変更したい

保存先(スキーマ案)

いまの schema には専用カラムが無いので、入れるなら以下のどれか。

案方式メリット / デメリット
A(簡単) documents に tags TEXT カラム追加 → JSON文字列で保存
例: '["レジ","認証サーバー"]'		 実装が一番楽。LIKE 検索でも引ける。
厳密な絞り込みクエリはやや書きづらい。
B(きれい) document_tags 関連テーブルを新設(document_id, tag)の many-to-many SQL で集計しやすい / インデックス効きやすい。
テーブルが増える / migration 必要。
C(暫定) 既存の chunks.metadata_json に詰める(今は固定で {"source":"seed"}) schema 変更不要。
本来 chunk 単位の付帯情報用なので意味が混ざる。

おすすめ: まず A で documents.tags を追加 → 運用が回って件数が増えてきたら B に移行。

検索でどう使うか

ベクトル検索だけでなく、tag が一致するドキュメントを優先するハイブリッド検索にする。
○ 流れの例
質問: 「レジの認証サーバーにログインできません」
① 質問文から tag 候補を抽出 → ["レジ", "認証サーバー", "ログイン不可"]
② documents.tags が↑のどれかにヒットする document に絞り込み
③ 絞り込んだ範囲で chunks をベクトル検索
④ tag ヒット数で score にボーナス(例: +0.1 / 一致 tag)
狙い: ベクトル類似度だけだと「ログイン不可」全般がヒットしてしまうが、tag で「レジ」に絞れるので答えが正確になる。

1つの問い合わせをルール通りに変換した例

「[顧客名]さんからのログイン不可の問い合わせ」を、ルールに沿って document + chunks にしてみる。

① 元の問い合わせ(PIIあり)

RAW(raw_inquiries.raw_text に保管 / 検索対象外)
「[顧客名]([EMAIL], [PHONE])です。
昨日からログインできません。パスワードを再設定しても入れません。
アカウントがロックされているかもしれません。確認お願いします。」

② PIIマスキング

sanitized(raw_inquiries.sanitized_text)
「[NAME]([EMAIL], [PHONE]) からログイン不可の問い合わせ。
パスワード再設定後もログインできない。アカウントロックの可能性あり。」

③ AI要約 → JSON

AI出力(これを documents / chunks に流す)

テーブル入る値

chunks { id="chunk_ext_001_1", document_id="doc_ext_001", chunk_index=1, content="パスワード再設定後も…確認する。" }

チェックリスト(これだけ守ればOK)

chunks を書くとき

  • 条件(〜の場合は)と対応(〜する)を同じ chunk に入れたか？
  • 長さは 50〜200字くらいに収まっているか？(1〜3文)
  • 個人情報(氏名・メール・電話・住所など)が入っていないか？
  • chunk 単体で読んで意味が通るか？(前後の chunk を読まないと分からない、になっていないか)
  • chunk_index は document 内で 1 から連番になっているか？

documents を書くとき

  • 1 document = 1トピックに絞れているか？(認証関連全般、みたいになっていないか)
  • title は「対象 + 操作 / 問題」の形になっているか？(〜の対応 / 〜の手順)
  • category は決められた enum から選んでいるか？(自由記述していないか)
  • visibility / status / source_type も決められた値から選んでいるか？

2026年版 RAG ベストプラクティス(Web資料からの追補)

このページで書いてきたルールが、業界の最新動向と合っているかを Web 資料(2026年版)で確認した結果のまとめ。
結論: 「タグ/メタデータでの絞り込み」「セマンティック分割」「ハイブリッド検索」 が現在の主流。今のルール案は方向性として合っているが、いくつか強化ポイントあり。

① チャンク戦略 — 単純な「句点で割る」だけだと弱い

戦略内容適性
Fixed-size 決まったトークン数で機械的に分割(例: 512トークン) FAQ・短い均質な文書には十分。長文には不向き
Recursive 段落 → 文 → 句、と階層的に分けて目標サイズに収める 汎用的でバランス良し。LlamaIndex などの標準
Semantic 文をベクトル化し、意味の切れ目で分割する 2026年 Kapa.ai ベンチで fixed比 +18% の検索精度。長文プロセに有効
Hierarchical 「小さいチャンクで探す → 親の大きいチャンクを LLM に渡す」の2階層 2025-2026 で最も普及している本番パターン。精度と文脈量を両立
Structure-based マニュアルの見出し階層(章・節)に沿って分割 このプロジェクト(手順マニュアル)に最も合う
このプロジェクトへの示唆: 「句点で1手順=1チャンク」は Structure-based の簡易版。長期的には「親 document = 大チャンク / 子 chunks = 細チャンク」の Hierarchical 構造に寄せる価値あり(検索は小チャンクで、AI に渡すのは親文書全体)。

② チャンクの先頭に「見出しパス」を入れると検索精度が跳ねる

○ Web資料で「最高 ROI の改善」と言われている手法
content: 「[認証 > ログイン不可 > パスワード再設定後] 再設定リンクの有効期限と発行履歴を確認する。」
効果: chunk 単体でも「どの章のどの節か」がベクトルに乗るので、似た文面の他チャンクと混ざらない。
実装: chunks.content に保存する文字列の先頭に [category > document.title > section_title] を付けるだけ。数行で実装できる。

③ オーバーラップ(隣接チャンクの重複)について

従来の定石: 10〜20% のオーバーラップを入れる(500トークンなら 50-100 トークン分を次のチャンクと共有)。
ただし 2026年1月の体系的調査では「オーバーラップは測定可能な精度向上をもたらさず、インデックスコストだけ増える」という結果も出ている。
→ このプロジェクトのように「1手順 = 1完結チャンク」なら オーバーラップなしで OK。

④ メタデータ(タグ含む)でのフィルタリングは検索の前段に置く

2026年の定説: ベクトル検索の 前に メタデータでフィルタして候補を絞る → 残った中でベクトル類似度を計算する。
(例: 「engineering 部門の質問」なら engineering 文書だけに絞ってからベクトル比較)
✗ ベクトル検索だけ
質問「レジが認証エラー」
→ 全 chunks にベクトル比較
→ 「Web管理画面の認証エラー」も同じくらいヒット
○ メタデータ前段フィルタ
質問「レジが認証エラー」
→ tags に "レジ" を含む documents に絞る
→ その範囲だけでベクトル比較
→ ノイズが消えてレジ向け回答だけが残る

⑤ ハイブリッド検索 — ベクトル + キーワード(BM25) を併用

ベクトル検索は「意味が似てる」、BM25(キーワード一致) は「単語が一致する」。両方の良いとこ取りをするのが ハイブリッド検索。

段階処理このプロジェクトでの将来実装
1. フィルタ tags / category / status でドキュメントを絞る documents.tags の LIKE / JSON 検索
2. ベクトル検索 クエリのベクトルと chunk のベクトルで cosine 類似度 既に実装済み(src/m05_api/m05_api_search_text.rs)
3. キーワード検索 BM25 や LIKE でクエリ語と chunk.content をマッチング SQLite FTS5 を使えば追加できる
4. 結果統合 Reciprocal Rank Fusion (RRF) で順位を合成 順位の逆数を足し合わせるだけ。実装は数十行
5. 再ランキング 上位 N 件を cross-encoder などで再評価 API 経由(任意)。精度を最後に詰める

⑥ メタデータは AI に自動生成させるのが定番化している

チャンクを LLM に渡して「{topic, category, tags, time_period}」を JSON で吐かせるパイプラインが標準化。
このプロジェクトの「PII マスキング → AI 要約 → JSON 化 → 投入」構想はこの流れと完全に一致している。方向性は合っている。

⑦ まとめ: このプロジェクトのルール案に追加で取り入れたい3つ

  • (a) チャンク本文の先頭に「見出しパス」を埋め込む ─ 例: [認証 > ログイン不可] 顧客が…。最小コストで効く改善。
  • (b) documents.tags を追加してメタデータフィルタを実装 ─ ベクトル検索の前段で絞り込む。tag揺れ防止リストを運用ルールで決める。
  • (c) BM25 検索(SQLite FTS5)を追加して RRF でベクトル検索と統合 ─ 固有名詞(製品名・ID)に強くなる。誤検索が減る。
