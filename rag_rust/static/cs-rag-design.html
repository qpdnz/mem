1. このツールが「何をする」か(再定義)

主役はオペレーター。顧客対応中に「過去どう答えたか / マニュアルにあるか」を キーボード数回で引けるのがゴール。
顧客の問い合わせを自動取り込みする仕組みは 本ツールの目的ではない(ナレッジ追加は別経路でいい)。

主機能

機能用途備考
横断検索FAQ / 過去事例 / マニュアル / 内部メモを一発検索ハイブリッド検索(キーワード+ベクトル+タグ絞り込み)
検索履歴自分の過去検索を一覧 / 再実行 / シェア本ツールの核。検索の組み立てを資産にする
有用度フィードバック「この結果が役に立った」「ハズレ」を1クリック順位学習と、運用評価の両方に使う
セッション1つの顧客対応 = 1セッション。複数検索をまとめる後から「この対応で何を引いたか」を辿れる
類似検索のサジェスト「他のオペレーターが似た問題でヒットさせた検索」暗黙知の共有
前の設計からの主な変更点:
・主役テーブルを raw_inquiries から search_sessions / search_queries に変更
・PII マスクは「顧客文を自動取り込む」のではなく 「オペレーターが検索窓に貼り付けたテキスト」を対象に行う
・documents への自動投入パイプラインは「あれば嬉しい」のオプション扱いに格下げ

2. オペレーターのユーザストーリー

典型シナリオ

1 顧客チケットを開く(別システム/別タブ)。「レジで認証エラーが出ます」と書かれている
2 本ツールを開き、検索窓に レジ 認証 ログインできない と入れる(または顧客本文をコピペ)
3 必要なら左ペインで tag フィルタ(「レジ」「認証サーバー」)、category(認証) を絞る
4 上位結果(documents → 該当 chunks)を見る。見出しパスと section_title で当たりをつける
5 役に立った結果に「👍 採用」、外れた結果に「👎 ハズレ」をクリック
6 同じセッション内で追加検索(例: ロック解除 手順)。検索履歴にひも付く
7 対応完了。セッションに「クローズ」フラグ。後から検索履歴を辿れる

「やりたい」ユースケース

ユースケース本ツールで提供する手段
過去に自分が引いた検索をもう一度「自分の検索履歴」一覧から再実行
先輩が同じ問題でどう引いたか見たい「類似クエリ」サジェスト + 公開された他者の検索履歴
1つの顧客対応の「引いた一覧」を引き継ぎセッションIDを共有(URL)
新人がよく外す検索のパターンを知りたい「👎多発クエリ」管理画面で集計
顧客文を貼ったら PII を勝手にマスクして検索してほしいクエリ前段の Sudachi+正規表現マスク

3. DB 設計図(検索履歴中心)

テーブルを役割で3層に分ける。
① 履歴層(主役) ／ ② 索引層(検索される側) ／ ③ 取り込み層(任意・オプション)。

① search_sessions 主役

  • id (PK)
  • operator_id
  • customer_ticket_ref (任意)
  • opened_at / closed_at
  • status (open / closed / abandoned)
  • title (任意, "レジ認証問題対応")

② search_queries 主役

  • id (PK)
  • session_id (FK→①)
  • operator_id
  • raw_query_text
  • sanitized_query_text
  • filters_json (tags/category/visibility)
  • query_embedding_json
  • executed_at

③ search_results 主役

  • id (PK)
  • query_id (FK→②)
  • chunk_id (FK→⑦)
  • document_id (FK→⑥)
  • rank
  • score_vector / score_bm25 / score_rrf

④ search_outcomes 主役

  • id (PK)
  • result_id (FK→③)
  • operator_id
  • verdict (helpful / not_helpful / used_in_reply)
  • comment
  • created_at

⑤ query_pii_masks PII

  • id (PK)
  • query_id (FK→②)
  • pii_type / placeholder
  • original_value (暗号化)
  • span_start / span_end

⑥ documents 索引

  • id (PK)
  • title / category / tags_json
  • visibility / status / source_type
  • created_by / approved_by

⑦ chunks 索引

  • id (PK)
  • document_id (FK→⑥)
  • chunk_index / section_title
  • content / content_with_header
  • token_count

⑧ embeddings 索引

  • id (PK)
  • chunk_id (FK→⑦)
  • embedding_model / version
  • embedding_json

⑨ chunks_fts 索引

  • SQLite FTS5 仮想テーブル
  • chunk_id, content_with_header
  • BM25 キーワード検索

⑩ tag_dictionary 索引

  • tag (PK)
  • group (product/operation/problem/...)
  • aliases_json(揺れ表記)
  • approved (bool)

⑪ knowledge_drafts 取り込み(任意)

  • id (PK)
  • source_kind (resolved_ticket / manual_paste / external_doc)
  • sanitized_text
  • suggested_summary_json
  • review_status
主役は ①〜④: 検索画面の操作はすべて履歴4テーブルに残る。本ツールの価値は「履歴を後で再利用できること」にある。
索引 ⑥〜⑩ は検索の対象: ベクトル/BM25/タグの3層検索を支える。実装は既存に近い。
取り込み ⑪ は任意: 「対応済みチケットからナレッジを起こす」専用フローはここに分離。MVPでは無くてもよい。

5. クエリ側マスク設計(Sudachi + 正規表現)

マスクの対象は「オペレーターが検索窓に入れたテキスト」。
オペレーターは顧客本文をコピペすることが多い。そのまま LLM や外部埋め込みAPI に渡すと PII 漏洩につながる。

5-1. Sudachi が拾うもの

PII種類検出方法備考
人名形態素解析(Mode.C)で品詞「名詞-固有名詞-人名」姓名分割される場合は連結処理
地名・住所「名詞-固有名詞-地名」+ 後続の番地/丁目/号NEologd / Sudachi 法人辞書併用推奨
企業名「名詞-固有名詞-組織」/「株式会社+固有名詞」業務上残したい場合はマスクOFF可のフラグを設ける

5-2. 正規表現が拾うもの

PII種類正規表現(参考)placeholder
メール[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}[EMAIL_N]
電話番号0\d{1,4}-?\d{1,4}-?\d{3,4}[PHONE_N]
クレカ\d{4}[ -]?\d{4}[ -]?\d{4}[ -]?\d{4}[CARD_N]
マイナンバー\d{4}\s?\d{4}\s?\d{4}[MYNUMBER_N]
注文IDORD-\d{8,}[ORDER_N]
会員IDMBR-\d{6,}[MEMBER_N]
生年月日/日付\d{4}[年/-]\d{1,2}[月/-]\d{1,2}日?[DATE_N]
郵便番号〒?\d{3}-?\d{4}[ZIP_N]

5-3. クエリ用パイプライン(オペレーターが Enter を押した瞬間に走る)

1 検索窓のテキストを受信(raw_query_text)
2 正規表現フェーズ: メール/電話/カード/ID/日付/郵便番号を先にマスク
3 Sudachi フェーズ: 残った本文を Mode.C で形態素解析、人名/地名/組織 をマスク
4 番号採番: 同セッション内では同じ値を同じ番号で使い回す([NAME_1] が複数クエリにまたがって維持される)
5 マスク済み本文を埋め込み API / FTS / DB 検索に流す
6 search_queries.sanitized_query_text と query_pii_masks に保存

5-4. 検索ワードの「正規化」も同時に行う

マスクとは別レイヤーで「揺れ吸収」をやる。クエリ→正規語に寄せると、tag マッチや BM25 のヒットが安定する。

入力正規化後仕組み
POS / レジ端末レジtag_dictionary.aliases_json で揺れ吸収
サインインログイン
2FA二要素認証
入れない / 入れませんログイン不可問題ラベル化(オプション)

7. 検索セッションサンプル × 4 件

各サンプルは ① セッション開始 → ② 検索クエリ(raw → masked → filters → 結果) → ③ outcome(役立ち評価) → ④ 履歴に残る形の4段。
オペレーターの「画面ストーリー」と「DB に何が残るか」が両方追える形にしている。

サンプル 1 / 認証 — レジで認証エラーの顧客対応
operator=op_007 / customer_ticket=TCKT-99812 / session=ses_001

① セッション開始

search_sessions INSERT
{ id="ses_001", operator_id="op_007", customer_ticket_ref="TCKT-99812", title=null, opened_at="2026-05-25T10:12:00+09:00", status="open" }

② クエリ 1 回目(顧客文をコピペ)

raw_query_text(検索窓に入った文)
「[顧客名]さん([EMAIL] / [PHONE])からの問い合わせ。店舗のレジ端末で認証サーバーにログインできない、パスワード再設定済みとのこと」
sanitized_query_text(マスク後 → 検索に使う)
「[NAME_1]さん([EMAIL_1] / [PHONE_1])からの問い合わせ。店舗のレジ端末で認証サーバーにログインできない、パスワード再設定済みとのこと」
name/email/phone を query_pii_masks に格納。検索本体は sanitized 側で実行。
filters(左ペインの絞り込み)
{ "category": "認証", "tags": ["レジ", "認証サーバー"], "visibility": "cs_only", "status": "approved" }
search_results(上位 3 件)

rank=2 chunk=chk_login_03 doc=doc_login (score_rrf=0.81) "[認証 > パスワード再設定 > 手順] 再設定リンクは発行から24時間有効…"
rank=3 chunk=chk_mfa_02 doc=doc_mfa (score_rrf=0.74) "[認証 > 多要素認証 > 失敗時] 失敗回数3回以上で…"

{ result_id=rank3, verdict="not_helpful", comment="今回は MFA 関係なかった" }

search_outcomes

セッションを後で開いたとき(引き継ぎ・自己レビュー)

4 サンプルの共通点:
・raw_query_text はオペレーターが実際に打った/貼った文(PII あり/なしどちらもある)
・sanitized_query_text がベクトル化・LLM 渡しの対象
・filters_json でカテゴリ/タグを絞ってから検索
・search_results の content_with_header でオペレーターが当たりを付けやすい
・search_outcomes の verdict が 本ツールの核資産(再ランキング学習 / FAQ 改善 / 新人サジェストに使う)

9. パイプライン全体図

1 オペレーターが検索窓に入力(顧客文の貼付 or 自分の語)
2 正規表現マスク(メール/電話/カード/ID/日付) → Sudachi マスク(人名/地名/組織)
3 検索ワードの揺れ正規化(tag_dictionary.aliases_json を適用)
4 sanitized_query_text を埋め込み API に投げてベクトル化
5 ハイブリッド検索: tag/category/status フィルタ → BM25(chunks_fts) + Vector(embeddings) → RRF で合成
6 上位 N 件を返す + search_queries / search_results / query_pii_masks に INSERT
7 オペレーターが 👍/👎/✅ をクリック → search_outcomes に INSERT
8 セッションを閉じると search_sessions.status=closed / closed_at 更新
9 集計: 「よく used_in_reply されるクエリ」「よく not_helpful 出る chunk」を管理画面で可視化 → ナレッジ改善ループ
本ツールの価値の生まれ方:
検索1回(②〜⑥)は普通の RAG ツール。履歴と outcome を毎回貯める(⑦⑧⑨)ことで、組織の検索ノウハウがデータ化される。
MVP は ②〜⑦ まで。⑧⑨ は後段の管理機能。
