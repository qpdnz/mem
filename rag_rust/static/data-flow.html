分割ルール早見表（元の文章 → どのテーブルの何カラムに入るか）

1本のマニュアル文書を「メタ情報」「本文の1手順」「ベクトル」の3粒度に手で分けて、別々のテーブルに入れています。
自動分割アルゴリズムはありません（src/m03_seed.rs に直書きしているのが現状）。

将来構想: 生の問い合わせを PII マスキング → AI要約 → JSON保存する想定フロー

現状の seed は人間が直接 title / chunks に整形済みの文を書いているが、本番運用では「人間が書くタイトルが適切とは限らない」「生問い合わせに個人情報が混じる」という前提が必要。
そのため、以下の流れで raw_inquiries → documents / chunks へ変換する構想を持っておく。

① 顧客からの生問い合わせ（PII を含む）
例: 「[顧客名]([EMAIL], [PHONE])です。ログインできません。パスワードを再設定しても入れない…」
→ raw_inquiries.raw_text にそのまま保存（RAG検索対象外の隔離領域）
② PII マスキング: 氏名・メール・電話番号などを検出して伏字化
「[NAME](([EMAIL], [PHONE]))です。ログインできません。パスワードを再設定しても入れない…」
→ raw_inquiries.sanitized_text / detected_pii_json に保存
③ AI要約: サニタイズ後の文を LLM に渡し、タイトル・分類・本文(1手順ごと)を JSON で出力させる
{
  "title": "パスワード再設定後もログイン不可の対応",
  "category": "認証",
  "chunks": [
    "再設定リンクの有効期限と発行履歴を確認する。",
    "アカウントロック・多要素認証の失敗回数を確認する。"
  ]
}
→ 人が書いたタイトルに依存せず、検索しやすい形に正規化される
④ 上記 JSON を分解して既存ルール A/B/C と同じく documents / chunks / embeddings に流し込む
・JSON.title → documents.title
・JSON.category → documents.category
・JSON.chunks[i] → chunks.content（chunk_index = i+1）
・各 chunk → embeddings に5次元ベクトル化して JSON 文字列で保存
狙い:
・生 PII を RAG 検索対象（chunks）に絶対に流さない（隔離は raw_inquiries だけで完結）
・人間がつけたブレのあるタイトルを AI が正規化し、検索ヒット率を安定させる
・要約後の JSON を保存しておけば、再投入・再ベクトル化が決定論的にできる（生文の再 AI 推論が不要）
未実装: ②③④ のいずれも現状コードにはなし。raw_inquiries / rag_queries はテーブル定義のみで INSERT 実装は存在しない（src/m02_db.rs）。

事前投入データの変換過程（マニュアル）

ここは「seed時に最初から入れているデータ」が、どの順でどの形に変換されるかを説明しています。

1. 文書メタを documents に登録
例: doc_001 / 「請求書の宛名変更対応」 / status=approved
2. 本文を chunks.content として登録
例: chunk_001 = 「顧客から請求書の宛名変更を依頼された場合は...」
3. ベクトルを作成して embeddings.embedding_json に保存
例: chunk_001 -> [0.91, 0.05, 0.02, 0.01, 0.0]
